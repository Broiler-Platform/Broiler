## 2. Google Search Compliance Context

### 2.1 What "Google Search Compliance" Means for Broiler

Google Search compliance, in the context of Broiler, means that the browser's
rendering engine produces output **functionally equivalent to what Googlebot
sees** when it crawls and renders a web page. Google's crawler uses a **Web
Rendering Service (WRS)** based on headless Chromium to execute JavaScript and
render pages before indexing them. A compliant Broiler rendering should:

- Execute the same JavaScript that WRS would execute, without uncaught
  exceptions that break the page's initialization chain.
- Apply CSS rules consistently with how Chromium applies them.
- Produce a visual output where all major content elements (text, images,
  interactive controls) are present and positioned correctly.

This matters because Broiler aims to be a fully functional web browser, and
Google Search is one of the most widely used pages on the web. Rendering it
correctly is a meaningful benchmark for HTML, CSS, and JavaScript fidelity.

### 2.2 How Google's Web Rendering Service (WRS) Works

Google's WRS is the rendering component of Googlebot. Key characteristics:

- **Engine**: Evergreen Chromium (always up-to-date with the latest stable
  Chromium release).
- **JavaScript**: Full ES2020+ support, including `async`/`await`, modules,
  `Promise`, and modern DOM APIs.
- **Rendering model**: WRS renders the page after executing JavaScript, then
  indexes the final DOM state. This means content generated by JS is indexed.
- **Stateless**: WRS does not persist cookies, localStorage, or session state
  between crawls. Each page load starts with a clean state.
- **Timeouts**: WRS has internal timeouts for script execution. Long-running
  scripts may be terminated before completion.
- **Viewport**: WRS uses a viewport of approximately 411 × 731 CSS pixels
  (mobile-first) or 1024 × 768 for desktop rendering.

For the purposes of this roadmap, our reference comparison uses a **desktop
viewport of 1024 × 768** to match our Chromium reference screenshots.

### 2.3 Key Google Guidelines Relevant to Broiler

The following official Google guidelines inform the compliance requirements
tracked in this roadmap:

| Guideline Area                        | Relevance to Broiler                          |
|---------------------------------------|-----------------------------------------------|
| JavaScript SEO basics                 | Broiler must execute JS to produce final DOM   |
| Fix lazy-loaded content               | Image loading and `IntersectionObserver`       |
| Understand the JavaScript SEO basics  | `document` / `window` API completeness         |
| Rendering on Google Search            | CSS and layout fidelity expectations           |
| Mobile-friendly test                  | Viewport and responsive rendering              |
| Lighthouse / Web Vitals               | `performance.now()` and timing API support     |

See [References](#references) for direct links to official documentation.

## 4. Image Comparison Methodology

The comparison pipeline follows the same approach used for Acid2 and Acid3:

1. **Broiler render**: `dotnet run --project src/Broiler.Cli --
   --capture-image https://www.google.com --output google-broiler.png
   --width 1024 --height 768`
2. **Chromium reference**: Playwright-based Chromium headless screenshot at
   identical viewport (1024 × 768)
3. **Pixel comparison**: Python tool (`scripts/google-compare.py`) produces:
   - Colour-coded diff image (`google-diff.png`)
   - Structured report (`google-report.txt`)
4. **Region-based analysis**: Content presence checked in:
   - Top bar (y=0–50): Gmail, Images, Sign in
   - Logo area (y=150–310): Google logo
   - Search box (y=310–380): Text input field
   - Buttons (y=380–440): Submit buttons
   - Footer (y=700–768): Footer links

## 6. JavaScript Exception Investigation

### 6.1 Exception 1: `Cannot get property mei of undefined`

**Stack trace origin**: `inline-vm.js:1,2` → `CaptureService.ExecuteScripts`

**Analysis**: Google's first inline script initialises a global `_g` object with
properties like `kEI`, `kEXPI`, `kBL`, `kOPI`, then assigns it to `window.google`.
A subsequent script tries to access a property `mei` on an object returned by a
DOM query or window property that resolves to `undefined` in Broiler.

**Suspected root cause**: Google's scripts call
`document.getElementById('some-id').mei` where the element doesn't exist in
Broiler's DOM (either the element was created by a previous script that failed,
or it relies on server-side rendering that produces elements Broiler doesn't see).

**Required fix**: Ensure `document.getElementById()` returns `null` (not
`undefined`) for missing elements, matching the DOM spec. Also consider adding
a `performance.now()` polyfill since Google's metrics code depends on it.

### 6.2 Exception 2: `Cannot get property addEventListener of undefined`

**Stack trace origin**: `inline-vm.js:6,5` → `inline-vm.js:1,28`

**Analysis**: A script calls `.addEventListener()` on the result of a DOM query
that returned `undefined`. This is a cascading failure — the element lookup
returns `undefined` instead of `null`, and the code doesn't guard against it.

**Suspected root cause**: Two possible issues:
1. `document.querySelector()` or `document.getElementById()` returning
   `JSUndefined` instead of `JSNull` for missing elements.
2. Missing DOM element created dynamically by an earlier script that failed.

**Required fix**: Audit all DOM query methods in `DomBridge` to ensure they
return JavaScript `null` (not `undefined`) when no element is found.

### 6.3 Exception 3: `Cannot get property clientWidth of undefined`

**Stack trace origin**: `inline-vm.js:1,56`

**Analysis**: Google's scripts access `element.clientWidth` to determine viewport
dimensions for responsive layout. The element lookup returned `undefined`.

**Suspected root cause**: Same DOM query null-vs-undefined issue as above.
Additionally, `clientWidth`/`clientHeight` may not be implemented on Broiler's
DOM element proxies.

**Required fix**:
1. Fix null-vs-undefined for DOM queries (shared root cause).
2. Implement `clientWidth`, `clientHeight`, `offsetWidth`, `offsetHeight`,
   `scrollWidth`, `scrollHeight` on DOM element proxies (return viewport
   dimensions or computed layout dimensions).

### 6.4 Impact Assessment

The three exceptions are **cascading failures** from a common root cause:
DOM element queries returning `JSUndefined` instead of `null`. This causes
every subsequent property access on the result to throw, breaking Google's
initialization chain. Fixing the null-vs-undefined issue would likely resolve
all three exceptions and allow Google's scripts to complete initialization.

## References

Official Google documentation relevant to this roadmap:

1. **Google Search Central — JavaScript SEO Basics**
   https://developers.google.com/search/docs/crawling-indexing/javascript/javascript-seo-basics
   — Explains how Google crawls, renders, and indexes JavaScript-powered pages.
   Directly relevant to Sections 6 and 8 (JS exceptions and DOM API coverage).

2. **Google Search Central — Fix Lazy-Loaded Content**
   https://developers.google.com/search/docs/crawling-indexing/javascript/lazy-loading
   — Guidelines for lazy-loading images and content. Relevant to TODO-G5
   (external image fetching) and TODO-G10 (`IntersectionObserver` polyfill).

3. **Google Search Central — Rendering on Google Search**
   https://developers.google.com/search/docs/crawling-indexing/rendering
   — Describes how WRS (Web Rendering Service) renders pages using headless
   Chromium. Key context for Section 2 (compliance context) and Section 4
   (image comparison methodology).

4. **Google Search Central — Googlebot Overview**
   https://developers.google.com/search/docs/crawling-indexing/googlebot
   — Describes Googlebot's capabilities, user-agent strings, and crawling
   behavior. Background for understanding what APIs and features WRS supports.

5. **Web.dev — Web Vitals**
   https://web.dev/articles/vitals
   — Defines Core Web Vitals metrics (LCP, FID, CLS) that depend on browser
   APIs like `performance.now()`. Relevant to TODO-G2 (performance API polyfill).

6. **MDN Web Docs — DOM API Reference**
   https://developer.mozilla.org/en-US/docs/Web/API/Document_Object_Model
   — Authoritative reference for DOM APIs that Broiler needs to implement.
   Use for verifying correct return values (e.g., `null` vs `undefined`)
   in Section 6 and TODO-G1.

7. **MDN Web Docs — Element: clientWidth property**
   https://developer.mozilla.org/en-US/docs/Web/API/Element/clientWidth
   — Specification for `clientWidth`/`clientHeight` behavior. Relevant to
   TODO-G4.

8. **WHATWG HTML Living Standard — Scripting**
   https://html.spec.whatwg.org/multipage/scripting.html
   — Specification for `<script>` element processing, including `async` and
   `defer` attributes. Relevant to TODO-G20 (async script handling).
